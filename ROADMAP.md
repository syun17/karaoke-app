# カラオケアプリ 開発ロードマップ（自力開発版）

## 目的と方針

最終目標は、マイクの声をリアルタイムに音程バーへ反映し、音源再生・歌詞表示・端末ローカルのライブラリを備えたカラオケアプリ。配布先は Web（データ保持なし）・exe・Android（オフラインでライブラリ有効）。

- コードは自分で書く。AI には「道しるべ」と「厳格レビュー」だけを頼む。
- 一番難しい **リアルタイム音程検出を最初に作り、遅延の体感を早期に検証する**。
- 各フェーズは必ず「動くもの」を出す。動かないまま次へ進まない。
- こまめに git コミット。1 フェーズ = 複数の小さなコミット。
- 一度に 1 機能だけ。仕様を勝手に広げない。

> 既存のプロトタイプは「目指す挙動（見た目・遅延感）の参照用」であって、コードを写すためのものではない。各フェーズは自分の手で書き直す。

## 使う技術（触れていく順番）

HTML / CSS / TypeScript → Web Audio API → Canvas → React（UI のみ）→ IndexedDB → Tauri（exe）→ PWA（Android）

## フェーズ 0: 土台づくり

- 学ぶこと: Node.js、Vite、TypeScript の基本、Git。
- 作るもの: Vite + React + TypeScript の空アプリ。
- 完了条件:
  - `npm run dev` で初期画面がブラウザに出る。
  - TypeScript の型エラーが出ない設定になっている。
  - git リポジトリを作り初回コミット済み。

## フェーズ 1: コア＝リアルタイム音程検出（最重要・最初にやる）

> ここが UX を決める。**React を使わず、素の TypeScript モジュール**として作る。

- 学ぶこと: `getUserMedia` / `AudioContext` / `AnalyserNode`、自己相関によるピッチ検出、Hz→音程（セント）変換、`requestAnimationFrame`、Canvas 描画。
- 作るもの（小さく段階的に）:
  1. マイク許可 → 音量（RMS）を数値表示。
  2. 自己相関で基本周波数（Hz）を検出して表示。
  3. Hz を音名＋セントに変換して表示。
  4. Canvas にカラオケ式の横スクロール音程バー（半音スナップ＋ヒステリシス）。
- 完了条件:
  - 声を出すと、半音にスナップしたフラットな棒が右から左へ流れる。
  - 解析窓サイズを変えると遅延の体感が変わることを自分で確認できる。
  - 無音・雑音でバーが暴れない（ノイズゲートが効く）。
- 詰まりやすい点: バッファ長と遅延のトレードオフ、オクターブ誤検出。

## フェーズ 2: React 化と画面・操作

- 学ぶこと: React の基礎（コンポーネント、props、state、`useState` / `useEffect` / `useRef`）。
- 重要原則: **フェーズ1のエンジンを React の再描画サイクルの外に置く**。毎フレームのピッチを `useState` に入れない。`useRef` でエンジンを保持し、rAF と Canvas 直書きで回す。
- 作るもの: エンジンを React アプリに組み込み、開始/停止・解析窓・ノイズゲートの操作 UI を React で作る。
- 完了条件:
  - フェーズ1と同じなめらかさのまま React 上で動く。
  - 操作 UI が効き、再レンダリングでカクつかない。

## フェーズ 3: 音源プレイヤーと歌詞

- 学ぶこと: `<audio>` 要素、Object URL、ファイル入力、キーボード/ホイールイベント。
- 作るもの:
  - 音源アップロード → 再生/一時停止・シーク・音量・時間表示。
  - 歌詞: テキストを行に分割し、前/現在/次の 3 行表示。↑↓ キーとスクロールで移動。
- 完了条件:
  - 単発で音源と歌詞を読み込み、再生しながら歌詞を手動で送れる。
  - 音源と声の判定（比較）はしない。
- 詰まりやすい点: Object URL の解放（メモリリーク）、シークと `timeupdate` の競合。

## フェーズ 4: ライブラリとストア抽象化

- 学ぶこと: IndexedDB、インターフェース設計、ビルド時フラグ（Vite の env）。
- 作るもの:
  - `LibraryStore` インターフェースを定義し、`IndexedDbStore` を実装。
  - ライブラリ画面（登録/選択/編集/削除）。曲名・アーティスト・音源・歌詞をセットで保存。
  - プラットフォーム判定で、Web ではライブラリを無効化（＝単発アップロードのみ）。
- 完了条件:
  - exe/モバイル想定ビルドでは曲を保存・一覧・読込でき、再起動後も残る。
  - Web 想定ビルドではライブラリ画面が出ず、サーバにも何も送らない。
- 詰まりやすい点: `file://` での IndexedDB 不可（localhost で開く）、保存対象は Blob。

## フェーズ 5: パッケージング（exe / Android）

- 学ぶこと: Tauri（exe 化）、PWA（manifest / service worker）、`navigator.storage.persist()`。
- 作るもの:
  - Tauri でデスクトップ exe をビルド（ライブラリ有効）。
  - PWA 化して Android にインストール可能・オフライン動作（ライブラリ有効、永続化を要求）。
  - （任意・発展）Tauri の fs + SQLite プラグインで、音源を実ファイル＋メタデータ DB に保存する堅牢版を `LibraryStore` の別実装として用意。
- 完了条件:
  - exe が起動し、オフラインでライブラリが使える。
  - Android にインストールでき、機内モードでも音程バー・再生・歌詞・ライブラリが動く。

## フェーズ 6: 仕上げ

- 学ぶこと: AudioWorklet、YIN/MPM、エラーハンドリング、アクセシビリティ、簡単なテスト。
- 作るもの:
  - 検出を AudioWorklet（音声スレッド）へ移し、遅延と安定性を改善。
  - アルゴリズムを YIN/MPM に差し替えて検出精度を向上。
  - 例外処理、キーボード操作、軽いテストを整備。
- 完了条件:
  - 体感遅延がさらに縮み、長時間でも安定して動く。

## 自力開発のルール（自分への約束）

- 詰まったら、まず公式ドキュメントを読む。それでも進まなければ Claude に「リンクを出して」と頼む（解説文は求めない）。
- 動いたら必ずコミット。壊れたらすぐ戻せるようにしておく。
- コピペで動かさない。「なぜ動くか」を説明できる状態を保つ。
- フェーズの完了条件を満たすまで、次のフェーズに進まない。
